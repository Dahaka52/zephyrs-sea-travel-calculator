# План рефакторинга после аудита

Дата: 2026-03-06  
Проект: `zephyrs-sea-travel-calculator`

## 1. Цели

1. Стабилизировать расчёты и исключить расхождения между UI, API и чатом.
2. Уменьшить вероятность регрессий за счёт унификации логики.
3. Снизить нагрузку на `game.settings` и убрать дублирующие перерасчёты.
4. Подготовить кодовую базу к безопасному расширению (новые корабли, условия, формулы).

## 2. Область работ

Входит:

1. `scripts/calculator/*`
2. `scripts/ui/*`
3. `scripts/main.js`
4. `scripts/utils/helpers.js`
5. Документация и тестовый контур (минимум smoke + unit для ядра расчётов)

Не входит на первом проходе:

1. Полная миграция на ES-модули/Vite/TypeScript.
2. Редизайн интерфейса.
3. Изменение игрового баланса формул без отдельного ТЗ.

## 3. Приоритеты (из аудита)

P0:

1. Единый расчёт `ft/раунд` и устранение расхождения `*10` vs константа.
2. Удаление дублирования расчётов (единый путь вычисления результата).
3. Снижение штормов событий и лишних `game.settings.set`.

P1:

1. Согласование расчёта манёвренности между UI и API.
2. Нормализация поведения `openCalculator`/`toggleCalculator`.
3. Корректное сохранение/восстановление окна.

P2:

1. Удаление dead code и несогласованных fallback-веток.
2. Чистка ассетов и технического шума.

## 4. Этапы выполнения

## Этап A. Базовая стабилизация (P0)

Задачи:

1. Вынести единый метод расчёта результата перехода в `TravelCalculator`:
   `calculateTravelFromData(data)` или эквивалент.
2. Перевести `TravelCalculatorUI.calculate` и `calculateAndSend` на этот единый метод.
3. Исправить расчёт `ftPerRound` на `ZEPHYR_FT_PER_KNOT_PER_ROUND` везде.
4. Упростить обработчики событий UI:
   оставить один маршрут `onInput -> recalc -> debouncedSave`.
5. Ввести debounce сохранения (200-400 мс) для `lastInput`.

Критерии готовности:

1. В чате и в live UI одинаковые значения скорости/`ft/раунд`.
2. Нет повторных вызовов пересчёта на одно пользовательское действие.
3. Количество вызовов `game.settings.set` заметно сокращено.

## Этап B. Консистентность API/данных (P1)

Задачи:

1. Привести расчёт манёвренности к единому правилу для UI и API.
2. Перенести `lastInput` в `client` scope (или ввести явный режим: `client` по умолчанию).
3. Уточнить контракт API:
   `openCalculator` всегда открывает, `toggleCalculator` переключает.
4. При рендере применять сохранённые `width/height/top/left` последовательно.

Критерии готовности:

1. Одинаковые результаты при вызове через UI и `game.seaTravelCalculator`.
2. Пользователи не перезаписывают форму друг другу.
3. Поведение открытия окна предсказуемо.

## Этап C. Чистка архитектуры (P2)

Задачи:

1. Удалить неиспользуемые переменные/параметры и устаревшие комментарии.
2. Свести проверки fallback к явным ошибкам и валидируемым guard-блокам.
3. Минимизировать inline-стили в `ChatRenderer`/UI, вынести в CSS, где возможно.
4. Убрать дублирующиеся ассеты и зафиксировать единую папку ресурсов.

Критерии готовности:

1. Линейный, читаемый поток данных без неиспользуемых веток.
2. Меньше скрытых fallback-состояний.

## Этап D. Контроль качества

Задачи:

1. Добавить unit-тесты на:
   - `calculatePolarSpeed`
   - `calculateSpeed`
   - `calculateTurningRadius`
   - `calculateTravel` (оба режима)
2. Добавить smoke-чеклист для Foundry UI:
   - смена корабля
   - режим distance/time
   - ветер/волны
   - отправка отчёта в чат
3. Зафиксировать тестовые сценарии с ожидаемыми значениями.

Критерии готовности:

1. Критические вычисления покрыты автотестами.
2. Перед релизом выполняется повторяемый smoke-проход.

## 6. Риски и контроль

1. Риск: изменение формул ломает баланс кампании.
   Мера: на этапах A/B не менять формулы, только консистентность и поток данных.
2. Риск: регрессии UI из-за переподписки событий.
   Мера: сначала внедрить единый обработчик и сравнить старые/новые результаты на одном наборе сценариев.
3. Риск: несовместимость с уже созданными макросами.
   Мера: сохранить публичные имена API и добавить мягкие совместимые обёртки.

## 8. Definition of Done

1. Устранены P0/P1 расхождения из аудита.
2. UI/API/чат дают одинаковые численные результаты для одинакового ввода.
3. Поведение окна и сохранения формы стабильно в многопользовательском мире.
4. Есть минимальный автотестовый контур для критических расчётов.
5. README и план актуализированы под финальную архитектуру.
